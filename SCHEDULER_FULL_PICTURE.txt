SCHEDULER FULL PICTURE (IDIOT-PROOF VERSION)
===========================================

This file explains the CURRENT scheduler behavior after the single-worker queue + timeout refactor.


1) QUICK GLOSSARY
-----------------
- Schedule: rule for when a MED file should run.
- Due: current time >= schedule start_time.
- Queue: waiting list of due jobs.
- Worker: one background runner that consumes queue items one-by-one.
- Execution record: one run attempt saved in SQLite history.
- Manual recovery: emergency state that pauses dispatch until user resolves.
- Timeout config: overdue policy with action (`continue` or `run_cleanup_and_terminate`).


2) THE LAYERS (WHAT EACH PART DOES)
-----------------------------------
Layer A - UI (React)
- User creates/edits/deletes schedules.
- Shows runtime queue details (`running`, `queued`, `waiting_reason`).

Layer B - API (FastAPI)
- Validates request, checks permissions, saves/loads data.
- Create path rejects past start_time.

Layer C - Storage (SQLite)
- Stores schedules, timeout config, and execution history.

Layer D - Scheduler Engine
- Wakes up every interval, finds due schedules, enqueues jobs.
- Owns dispatch gating (maintenance/manual recovery/busy checks).

Layer E - Single Worker Queue
- Exactly ONE worker pulls jobs in order.
- No scheduler-side slot-retry layer.

Layer F - Executor (HxRun launcher)
- Runs pre-execution steps for normal runs.
- Launches HxRun command.
- Applies timeout action at launch time (including queue-wait delay effects).

Layer G - Recovery/Notifications
- Sends notifications.
- Can enforce manual recovery after abort-like failures.


3) MAIN MENTAL MODEL
--------------------
Database stores "what should run".
Scheduler enqueues "what is due now".
Single worker waits until dispatch gates are clear.
Worker launches executor once ready.


4) WHAT HAPPENS AT 10:00 (NORMAL CASE)
--------------------------------------
Example: Demo.med, daily, due at 10:00.

- 10:00 scheduler loop sees job is due.
- Job is added to in-memory queue.
- Worker picks it when idle.
- If gates are clear, execution moves to `running` and launches HxRun.
- If success: execution marked `completed`, next run moved forward.


5) WHAT HAPPENS WHEN MANY JOBS ARE DUE TOGETHER
------------------------------------------------
Example: A, B, C all due at 10:00.

- Scheduler enqueues A, B, C.
- Worker runs A first.
- B and C stay queued.
- After A ends, worker runs B, then C.

No parallel scheduler workers are created.


6) DISPATCH GATES (WHY QUEUED JOBS MAY WAIT)
---------------------------------------------
Before launching, worker checks gates repeatedly. If blocked, job stays queued with `waiting_reason`.

Current blocking reasons:
- HxRun maintenance mode is enabled.
- Global manual recovery is active.
- Schedule itself is marked `recovery_required`.
- Hamilton HxRun process is currently busy.

If a queued schedule is removed/deactivated before dispatch, execution is marked `cancelled` with a reason.


7) TIMEOUT POLICY (CURRENT, NO RETRYCONFIG)
--------------------------------------------
There is NO schedule retry policy (`RetryConfig` was removed from scheduling).

Each schedule can have `timeout_config`:
- `timeout_minutes` (optional)
- `action = continue` (default)
- `action = run_cleanup_and_terminate` (requires cleanup method path)

How timeout is used:
- Timeout is evaluated from scheduled start_time + timeout_minutes.
- Evaluation happens at launch time too, so queue waiting counts toward lateness.
- If timed out + `continue`: run normal method.
- If timed out + `run_cleanup_and_terminate`: run cleanup method instead, skip normal pre-execution pipeline, and deactivate schedule for future runs.


8) HXRUN BUSY EXAMPLE (10:00)
-----------------------------
- 10:00 due -> queued.
- Worker tries dispatch, sees HxRun busy.
- Job remains queued (not running), with `waiting_reason` about busy HxRun.
- When HxRun becomes free, worker dispatches and runs job.


9) OVERDUE / MISS POLICY
------------------------
There is no lateness-based auto-miss now.

- If active and start_time <= now, schedule is due.
- Due jobs are enqueued even when late.
- One-time schedules are deactivated after successful completion.


10) START-TIME + PAYLOAD CONSTRAINTS
------------------------------------
- No duplicate-time rejection.
- No minute rounding in API persistence.
- Start-time precision is preserved (seconds are kept).
- Create endpoint rejects `start_time` in the past.
- Update endpoint keeps provided precision and does not apply minute conflict checks.


11) QUEUE STATUS API (RUNTIME-TRUTH)
------------------------------------
`/api/scheduling/status/queue` now reflects scheduler runtime queue state.

It includes:
- Running and queued counts.
- Detailed running/queued entries.
- `waiting_reason` for blocked queued items.

So UI queue cards reflect the real worker pipeline.


12) EXECUTION STATUS VALUES
---------------------------
Scheduling execution records now use:
- `pending`
- `running`
- `completed`
- `failed`
- `cancelled`


13) COMMON Q&A
--------------
Q: Can two schedules execute at the same time now?
A: Not through scheduler worker. One worker executes one job at a time.

Q: Is queue status based on real scheduler runtime?
A: Yes. `/status/queue` reads scheduler runtime queue snapshots.

Q: If HxRun is busy, does the schedule fail immediately?
A: No. It waits in queue until dispatch gates clear.

Q: Is retry behavior configured per schedule now?
A: No. Per-schedule retry config was removed; timeout action is used instead.


14) ONE-LINE SUMMARY
--------------------
Current flow: due-job detection -> queue -> single worker gate checks -> launch -> timeout action handling -> DB updates.
